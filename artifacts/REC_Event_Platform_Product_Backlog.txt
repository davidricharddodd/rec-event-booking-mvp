Product Backlog — Developer Edition with Vibe Code Briefs
Author: David Dodd, R |  Generated: June 2026  |  Source: Discovery Transcript + Spec v1.0
Overview
This backlog covers the REC Event Booking Platform — a custom-built replacement for Cvent, commissioned from an external vendor. The backlog was generated from the Event Booking System requirements specification (v1.0, January 2026), the vendor discovery and proposal transcript.
The Event Booking Platform backlog contains 18 new features across 10 epics: Event Architecture, Registration & Checkout, Payment Processing, CS5 Website Integration, Email Communications, Reporting & Dashboard, Mobile Applications (Staff + Attendee), Dynamics CRM Integration, AI Features, Data Migration, and GDPR/Security. The total estimated build timeline confirmed by the vendor is 21 weeks, with Figma designs in Weeks 1-4, development Weeks 5-16, and testing/deployment Weeks 17-21, targeting go-live end of July / early August.
The Video Transcription App backlog contains 1 bug fix, 5 new features, and 3 enhancements. The critical path for the transcription app is: FIX-01 (Mux static renditions), then FEAT-20 (upload), FEAT-21 (state machine), FEAT-22 (transcription), FEAT-23 (summarisation), FEAT-24 (Q&A). ENH-01 through ENH-03 are iterative improvements. Key open dependencies (detailed in the Dependencies table) include Dynamics 365 API access from London Systems, Cvent data export, Microsoft Entra ID SSO configuration, and REC App Store developer accounts.

Product 1 — REC Event Booking Platform
The following 18 feature cards cover the full scope of the custom-built Cvent replacement. Features are ordered by dependency — FEAT-01 through FEAT-03 (event architecture, templates, registration types) are foundational and must ship before FEAT-04 onwards. FEAT-12 (Dynamics integration) should be started in parallel from Week 5. FEAT-15 (Cvent migration) is on the critical path for go-live and must complete before Week 21.

  FEAT-01    [NEW FEATURE]    Standalone & Umbrella/Series Event Architecture
User Story
As a REC event manager, I want to create both standalone single-date events and umbrella/series events with multiple occurrences under one event page, So that I can manage REC's entire programme—from one-off conferences to recurring training courses—from a single platform.
Background & Context
The existing Cvent platform is being replaced with a fully owned custom solution. REC runs both individual events (conferences, seminars) and recurring training series. Karen confirmed in the discovery call that umbrella events need a single event page where customers select their preferred date.
Each date in a series must be independently trackable for capacity and attendance. The spec requires reporting to distinguish individual date performance from overall series performance. This feature is the foundation upon which templates (FEAT-02), registration (FEAT-03), and the mobile check-in app (FEAT-10) all depend.
Acceptance Criteria
✓  Event manager can create a Standalone event with a single date, time, location, and capacity.
✓  Event manager can create an Umbrella/Series event with multiple date instances, all displayed under one event URL.
✓  Each date within an umbrella event maintains its own: capacity limit, sold count, attendee list, and check-in records.
✓  Customer-facing booking page for an umbrella event shows all available dates with remaining spaces; customer selects preferred date during registration.
✓  Platform enforces no-overbooking: dates at capacity display as "Full" and cannot be selected.
✓  Admin reporting shows metrics both per-date (e.g., "Session 3 — 45/50 booked") and at series level (aggregate totals).
✓  Event type (Standalone or Umbrella) is displayed as a visible badge in the admin dashboard and event list.
✓  If the last date in an umbrella series passes, the event is automatically archived and removed from the public calendar widget.
Vibe Code Brief
Build the event data model in PostgreSQL. Events table: {id, type: enum(standalone, umbrella), title, description, category_id, template_id, status: enum(draft, published, archived), created_at}. Event_dates table: {id, event_id, start_datetime, end_datetime, venue_id, capacity, sold_count, status: enum(open, full, cancelled)}. POST /events creates event + dates in a single transaction. GET /events/:id returns event with dates array. On the Next.js event page, conditionally render a single date display (standalone) or a DateSelector component (umbrella) showing all dates as selectable cards with capacity badges. Do not modify payment or email flows in this sprint. Verify: create both types, confirm each date tracks capacity independently, confirm full dates show as unselectable.


  FEAT-02    [NEW FEATURE]    Event Template System
User Story
As an event manager, I want to save reusable event templates with pre-configured settings, So that I can create a standard event in under 5 minutes rather than re-entering all configuration from scratch.
Background & Context
Karen confirmed in the discovery call that having preset settings to "fast track event creation" is essential. The current Cvent workflow relies heavily on templates. Templates should capture all standard event configuration — branding, email sequences, registration types, and page layout.
The vendor confirmed template support with the ability to duplicate and update only dates, enhanced by the AI prompt feature (FEAT-14). Critically, editing a template must not retroactively alter events already created from it — each event must be independent once created.
Acceptance Criteria
✓  Event manager can create a named template capturing: event type, registration types, pricing defaults, capacity defaults, email sequences, branding settings, and page layout.
✓  Template library shows all available templates with name, last-used date, and a preview badge indicating event type.
✓  Creating a new event from a template pre-fills all form fields; manager only needs to supply date(s) and final title.
✓  A template can be duplicated to create a variant without modifying the original.
✓  Templates can be organisation-wide (all managers) or user-specific (private); scope is set at creation.
✓  Editing a saved template does not retroactively change events already created from it (event configs are snapshotted at creation).
✓  New event from template to "ready to publish" is achievable in under 5 minutes for a standard event.
✓  Deleting a template is blocked (with a warning) if any live future events were created from it.
Vibe Code Brief
Create a templates table: {id, name, type, config_json JSONB, scope: enum(global, user), created_by, created_at}. Config JSONB stores the complete event creation payload. POST /templates creates; GET /templates lists (filtered by scope=global OR created_by=currentUser); PUT /templates/:id updates (check no live events use it before allowing changes); POST /events/from-template/:id creates a new event pre-populated with template values — the event stores a snapshot of the config, not a live FK. Frontend: add a "Use Template" modal to the event creation page with a searchable template browser. On template select, populate all form fields and show a "template applied" banner. AI prompt duplication (FEAT-14) should call this endpoint internally. Do not connect AI in this sprint. Verify: create template, spawn two events, edit template, confirm neither event is changed.


  FEAT-03    [NEW FEATURE]    Registration Types, Tiered Pricing & Capacity Control
User Story
As an event manager, I want to define multiple registration types with independent pricing, capacity limits, and time-based pricing rules, So that REC can accurately apply membership-based pricing and prevent overbooking.
Background & Context
The spec requires support for registration types including Member, Non-Member, Student, Early Bird, and VIP — each with its own price and capacity. Karen also confirmed in the call that recurring events can have updated price books applied. Early bird cutoff dates and discount codes are explicitly required.
The Dynamics CRM integration (FEAT-12) will supply membership status in real time to validate which pricing tier a registrant qualifies for. This feature gates the basket and checkout flows (FEAT-04) as it determines what registrants can purchase.
Acceptance Criteria
✓  Unlimited registration types can be defined per event date, each with: name, price (in pence), capacity, visible_from, and visible_until dates.
✓  Admin can configure early bird pricing with an automatic cutoff date/time; standard pricing is shown once the cutoff passes.
✓  Discount and promotional codes can be created with: value (percent or fixed), max_uses, expiry date, and optionally scoped to specific events or registration types.
✓  Group booking: admin can configure a minimum group size that triggers an automatic percentage discount.
✓  Real-time capacity tracking: when a registration type reaches its limit, it displays as "Sold Out" and cannot be selected.
✓  Customer-facing booking form shows remaining spaces per registration type (e.g., "8 spaces remaining").
✓  When membership tier is retrieved from Dynamics CRM, the booking form automatically highlights eligible member-priced types.
✓  Admin can manually override a registration type's capacity limit from the event management screen; override is logged to the audit trail.
Vibe Code Brief
Create registration_types table: {id, event_date_id, name, price_pence, capacity, sold_count, visible_from, visible_until, is_member_only}. Discount_codes table: {id, code, discount_type: enum(percent, fixed), value, max_uses, uses_count, expires_at, event_ids JSONB}. POST /registration-types creates types for an event date. At checkout, call GET /membership/:email to retrieve tier from Dynamics CRM (FEAT-12 integration) and filter eligible types. Apply discount: POST /discount/validate accepts {code, basket_total} and returns {valid, discount_amount}. Capacity enforcement: use a SELECT FOR UPDATE on registration_types.sold_count when incrementing at checkout to prevent race conditions. Admin override: PUT /registration-types/:id/capacity-override requires admin role, writes to audit_log. Frontend: render registration types as radio cards with price, remaining spaces badge, and a "Member" tag where applicable. Verify: create early bird type, advance past cutoff, confirm standard pricing shown; test sold-out state.


  FEAT-04    [NEW FEATURE]    Multi-Event Basket & Single Checkout
User Story
As a customer, I want to add multiple events to a basket and complete payment in one transaction, So that I can book several REC events at once without re-entering my details and payment information for each one.
Background & Context
The spec mandates a persistent basket for logged-in users across browsing sessions. The vendor confirmed "a single checkout experience for RC customers." A single Stripe payment intent covers all basket items. Individual confirmation emails per event are required alongside a consolidated receipt.
Basket abandonment tracking and recovery emails are in scope per the spec. This feature depends on FEAT-03 (registration types) for pricing and FEAT-06 (Stripe) for payment processing. If a basket item's event sells out before checkout is completed, the customer must be notified clearly.
Acceptance Criteria
✓  Logged-in customers can add events (with selected date and registration type) to a persistent basket from any event page.
✓  Basket persists across browsing sessions (stored server-side, linked to user account).
✓  Basket sidebar/page displays: event name, date, registration type, price, and a remove button for each item.
✓  Single checkout flow processes all basket items in one Stripe PaymentIntent for the combined total.
✓  On successful payment, individual confirmation emails are sent per event plus one consolidated receipt email for the full transaction.
✓  If a basket item becomes sold out before checkout completes, the item is flagged with "No longer available" and checkout is blocked until the item is removed.
✓  Basket abandonment: if an active basket has items but no checkout within 24 hours, a recovery email is automatically sent.
✓  Order history in the customer account shows all individual bookings and the basket grouping they originated from.
Vibe Code Brief
Create baskets table: {id, user_id, status: enum(active, checked_out, abandoned), created_at} and basket_items: {id, basket_id, event_date_id, registration_type_id, price_pence_snapshot, status: enum(active, purchased, removed, sold_out)}. POST /basket/add validates availability first, then adds item. GET /basket returns current active basket. At checkout, create Stripe PaymentIntent for sum of all item price_pence_snapshots. On Stripe webhook payment_intent.succeeded: mark basket checked_out, create individual booking records, trigger per-event confirmation emails and consolidated receipt. Availability re-check at checkout start: if any item is sold out, return 409 with sold_out_items list. Abandonment: nightly cron marks baskets active for >24h as abandoned and queues recovery email via FEAT-07 email service. Frontend: floating basket icon with item count badge, slide-out drawer listing items, and "Checkout" button. Verify: add 2 events, checkout, confirm 3 emails sent (2 confirmations + 1 receipt).


  FEAT-05    [NEW FEATURE]    CS5 Website Event Calendar Widget (Embedded)
User Story
As an event manager, I want published events to automatically appear on the REC website calendar with working dropdown filters, So that the website is always accurate without requiring manual CMS updates or CS5 developer involvement.
Background & Context
The current CS5 integration is broken — the legacy API-driven dropdown filters no longer work and have been replaced with a plain text Cvent widget. Karen confirmed this as a pain point: making website changes for the event calendar requires CS5 intervention each time.
The vendor proposed a fully self-contained iframe component rendering independently of CS5 CMS logic. The widget handles all filtering internally so CS5 "doesn't need to worry about anything." This is the correct architecture: the platform renders a contained UI block that CS5 simply embeds with a one-line HTML snippet.
Acceptance Criteria
✓  Published events appear in the CS5 calendar widget within 5 minutes of being marked as Published in the platform.
✓  The widget renders inside an iframe embedded on CS5 with no dependency on CS5 CMS logic or rebuild steps.
✓  Customers can filter events by: category/type, date range, location, and availability status — all driven from within the iframe.
✓  Each event listing links directly to the event registration page on the event platform.
✓  Widget is fully responsive: correct display on mobile, tablet, and desktop.
✓  Widget branding (colours, fonts) matches REC brand guidelines and is configurable by the event manager without CS5 involvement.
✓  Unpublished events or events whose last date has passed are automatically removed from the widget.
✓  CS5 team can embed the widget using a single HTML snippet with no API keys in the frontend code (auth is on the served endpoint).
Vibe Code Brief
Build a GET /widget/events endpoint that returns published future events as JSON (no authentication required; events are public). Create a standalone Next.js page at /embed/calendar that is designed to be iframed — no navigation, headers, or footers. The page fetches from /widget/events on load and re-fetches every 5 minutes. Support URL query params: ?category=training&location=london&from=2026-01-01. Render a responsive card grid: event title, date chip, category badge, available-spaces indicator, Book Now CTA. Category filter widget reads available categories from GET /categories endpoint and builds a dropdown. On publish, the event status change triggers a cache invalidation for the /widget/events endpoint (use Next.js revalidatePath or a Redis cache bust). Provide CS5 team with: <iframe src="https://events.rec.uk.com/embed/calendar" width="100%" height="700" frameborder="0" style="border:none;"></iframe>. Verify: publish an event, confirm it appears in the iframe within 5 minutes; test all filter combinations.


  FEAT-06    [NEW FEATURE]    Stripe Payment Gateway Integration
User Story
As a customer, I want to pay for event registrations securely by card within the REC platform, So that I can complete my booking instantly without being redirected to a third-party payment site.
Background & Context
Stripe is the specified payment processor (confirmed in the spec Appendix A and in the vendor transcript: "the Stripe response will straight-away come into this platform" via webhooks). PCI compliance is non-negotiable. The system must support refunds, failed payment retries, and a full audit trail.
Critically, the platform must never store raw card data. All card input is handled by Stripe-hosted Elements (SAQ A scope). Payment confirmation is event-driven via webhooks — bookings are only confirmed once Stripe confirms payment, not when the user submits the form.
Acceptance Criteria
✓  Customers pay using Stripe card input (Stripe PaymentElement); no custom card form is built.
✓  Booking is confirmed only on receipt of a Stripe payment_intent.succeeded webhook — not on form submission.
✓  Failed payments display a clear error message; customer can retry with the same or a different card without losing basket contents.
✓  Admin can process full or partial refunds from the booking management screen; refund is initiated via Stripe API and reflected in the booking status immediately.
✓  All transactions are logged: event, amount (pence), Stripe PaymentIntent ID, status, timestamp, user ID.
✓  Platform stores only Stripe Customer IDs and PaymentIntent IDs — no raw card numbers or CVV data.
✓  Stripe webhook signature is validated on every incoming webhook request (stripe.webhooks.constructEvent).
✓  Payment status is visible on the event dashboard against each booking (Paid / Refunded / Pending / Failed).
Vibe Code Brief
Integrate Stripe using the stripe Node.js SDK and @stripe/stripe-js on the frontend. POST /payments/create-intent creates a PaymentIntent for basket total (pence) and returns client_secret. Frontend: render Stripe PaymentElement component — never build a custom card input. On Stripe webhook POST /payments/webhook: verify signature, handle payment_intent.succeeded (confirm booking, trigger emails, update basket), handle payment_intent.payment_failed (notify user). Refund: POST /admin/bookings/:id/refund calls stripe.refunds.create({payment_intent: booking.stripe_pi_id, amount: optional_partial_amount}). Store payment_records table: {id, basket_id, stripe_payment_intent_id, amount_pence, status, refund_amount_pence, created_at}. Ensure idempotency: check payment_records for existing PI ID before creating duplicate records on webhook retry. Verify: full payment flow, partial refund, failed card retry, webhook signature rejection.


  FEAT-07    [NEW FEATURE]    Automated Email Workflows
User Story
As an event manager, I want automated emails sent to registrants at key lifecycle points with personalised content and calendar invites, So that attendees are always informed without me sending manual emails for every booking.
Background & Context
The spec lists six required automated email types: registration confirmation (with .ics calendar invite), payment receipt, pre-event reminders, post-event follow-up, waitlist notifications, and cancellation confirmation. The vendor offered two email infrastructure options: use the existing Dynamics 365 email templates (preferred, as templates already exist) or build a native SMTP/SendGrid integration.
Both options require a flexible job queue approach where emails are scheduled at booking time (e.g., reminder = event_date - 7 days). Karen noted that personalisation tokens are essential. This feature depends on FEAT-06 (payment confirmation triggers the first emails) and optionally FEAT-12 (Dynamics integration for email delivery).
Acceptance Criteria
✓  Registration confirmation email sent immediately on payment confirmation, containing: event name, date/time, location, registration type, and an .ics calendar invite attachment.
✓  Payment receipt email sent as a separate message with itemised cost breakdown.
✓  Pre-event reminder emails sent at configurable intervals (admin-configurable per event; defaults: 7 days and 1 day before event).
✓  Post-event thank you email sent within 2 hours of event end date/time.
✓  Waitlist notification sent automatically when a space in the registrant's chosen type becomes available.
✓  Cancellation confirmation sent immediately on booking cancellation, including refund amount and timeline.
✓  All email templates support personalisation tokens: {first_name}, {event_name}, {event_date}, {registration_type}, {venue_name}, {booking_reference}.
✓  Email sending failures are logged; failed emails retry up to 3 times before being flagged for manual review.
Vibe Code Brief
Build an email_jobs table: {id, type: enum(confirmation, receipt, reminder, post_event, waitlist, cancellation), booking_id, send_at, status: enum(pending, sent, failed), attempts}. At booking confirmation, INSERT jobs for: confirmation (now), receipt (now), reminder_7d (event_date - 7 days), reminder_1d (event_date - 1 day), post_event (event_end + 2 hours). A background worker (cron every 2 minutes) selects pending jobs WHERE send_at <= NOW(), calls the email dispatch function, and updates status. For Dynamics email delivery: call the Dynamics 365 Email API using an existing template ID + substitution tokens. For .ics generation: use the ical-generator npm package — create an event with DTSTART, DTEND, SUMMARY, LOCATION, and attach as base64 to the confirmation email. For fallback/native SMTP: use nodemailer with the configured SMTP server. Log all sends to email_send_log table. Verify all 6 email types trigger and arrive with correct personalisation tokens.


  FEAT-08    [NEW FEATURE]    Real-Time Event Management Dashboard
User Story
As an event manager, I want a real-time dashboard showing all events, live booking counts, revenue, and a needs-attention panel, So that I can monitor event health at a glance without running individual reports.
Background & Context
The spec requires a calendar view, event status indicators, registration-vs-capacity counts, revenue per event, and a metrics summary. Karen currently uses the Cvent organiser app to check booking counts; this dashboard replaces that workflow and must be accessible from both desktop and tablet. The vendor confirmed "a real time dashboard for the booking system."
The dashboard is the primary admin landing screen. It must surface events needing attention (low registrations approaching, events approaching deadline) proactively. Full reporting detail is covered in FEAT-09; this feature is the live operational view.
Acceptance Criteria
✓  Dashboard default view is a calendar layout showing all scheduled events; switchable to a list/grid view.
✓  Each event card shows: event name, date(s), registered count / capacity (with a colour-coded progress bar), revenue total, and a status badge (Draft / Upcoming / Live / Past / Cancelled).
✓  Metrics row at top shows: total registrations today, this week, this month, and total revenue this month.
✓  "Needs Attention" panel surfaces: events with registrations below 50% of capacity within 14 days of event date, and events with outstanding unconfirmed payments.
✓  Clicking an event card opens a detail slide-out: full attendee list, payment status per attendee, and a "Email Attendees" button.
✓  Dashboard data refreshes automatically every 30 seconds without a full page reload.
✓  Filters allow narrowing by: event category, date range, event manager (admin role only), and status.
✓  Dashboard is usable on tablet (minimum 768px viewport) for use by staff during events.
Vibe Code Brief
Build GET /dashboard/summary API endpoint: returns event list with computed stats (sold_count, capacity, revenue_pence), summary metrics, and needs_attention array. Compute via PostgreSQL queries: JOIN event_dates with registration_types, bookings, and payments tables. For real-time updates: use a polling hook in React (useEffect with setInterval 30s) calling the summary endpoint; display a "Last updated: X seconds ago" indicator. Dashboard layout in Next.js: top row of 4 StatCard components, below that a CalendarView / ListView toggle, then EventGrid. Needs-attention logic: WHERE event_date BETWEEN NOW() AND NOW()+INTERVAL 14 DAYS AND sold_count/capacity < 0.5. Use Recharts for the capacity progress bars (LinearProgress equivalent). Detail slide-out: GET /events/:id/attendees with payment_status filter. Do not replace FEAT-09 reporting — this is operational view only. Verify: book a new attendee and confirm dashboard reflects the change within 30 seconds.


  FEAT-09    [NEW FEATURE]    Attendance, Revenue & Registration Reporting
User Story
As an event manager or finance team member, I want detailed reports on registrations, attendance, no-shows, and revenue with export capability, So that I can measure event performance and reconcile financial records.
Background & Context
The spec has a comprehensive reporting section covering revenue by event/date range/registration type, outstanding payments, refund history, and export to CSV/Excel/PDF. Karen noted that tracking people who booked but didn't attend is important — these are candidates for follow-up emails and feedback forms.
The finance team additionally requires a breakdown by payment method and a full refund history report. Reports must handle up to 10,000 registrations per query within 5 seconds. Report results should be cached to avoid repeated heavy DB queries.
Acceptance Criteria
✓  Event-level report shows: total registrations, checked-in (attended), no-shows, cancellations, total revenue, refunds, and net revenue.
✓  Cross-event report is filterable by: date range, event category, registration type, and event manager.
✓  Revenue report breaks down by: registration type, payment method, and individual event.
✓  No-show report lists individual attendees not checked in, with a one-click option to add them to a manual email segment.
✓  All reports export to CSV and Excel (.xlsx) with formatted headers.
✓  Refund history report includes: original booking date, refund date, amount, Stripe reference, and optional reason code.
✓  Registration trend chart shows weekly registration volumes for a selected event or date range.
✓  Reports load within 5 seconds for queries covering up to 10,000 registrations.
Vibe Code Brief
Create a /reports API module: GET /reports/events/:id/summary, GET /reports/events/cross?from&to&category, GET /reports/revenue?from&to, GET /reports/no-shows/:event_date_id, GET /reports/refunds?from&to. Each endpoint runs PostgreSQL aggregation queries. For performance: add composite indexes on bookings(event_date_id, status) and payments(created_at, status). Cache results for 5 minutes using node-cache or Redis. For Excel export: use the exceljs npm package to generate .xlsx with styled headers (bold, background colour). For CSV: use papaparse unparse. Frontend: tabbed report interface using React — Date range pickers (react-datepicker), filter dropdowns, a sortable data table (react-table), Export CSV button, Export Excel button. No-show report: checkbox per row, "Add to email segment" CTA. Trend chart: recharts LineChart with week-by-week registration data. Verify: cross-event report for 6 months of data, confirm no-show count matches manual count for a test event.


  FEAT-10    [NEW FEATURE]    Staff Check-In Mobile App (iOS & Android)
User Story
As a REC events staff member, I want to check in delegates on the day of an event using a mobile app, So that I can efficiently manage event arrivals without needing a laptop or access to the full web platform.
Background & Context
Karen confirmed that staff currently use Cvent's "On Arrival" app: see the attendee list, tap to check in, data syncs back to Cvent. This is the priority mobile feature — Karen stated staff check-in is "more urgent" than the attendee app. The app must support offline mode for venues with poor connectivity.
For most events, tap-to-check-in against a name list is sufficient. For the annual conference, QR code scanning is required (covered in FEAT-16). The vendor confirmed the app will be React Native, available on iOS App Store and Google Play. Staff and attendee views will be in the same app with role-based routing (FEAT-11).
Acceptance Criteria
✓  Staff log in with REC credentials (via Entra ID SSO — FEAT-17) and see a list of upcoming events assigned to their region.
✓  Selecting an event shows the attendee list for that specific date, searchable by name.
✓  Tapping an attendee's name row toggles check-in status (checked-in shown with a green tick; toggling reverses it).
✓  Check-in status syncs to the platform in real-time; checked-in counts appear on the web dashboard within 30 seconds.
✓  App functions in offline mode: check-ins are queued locally and sync automatically when connectivity is restored.
✓  App is published and available on iOS App Store and Google Play under the REC organisation developer accounts.
✓  Staff can add a walk-in attendee (name, email, registration type) directly from the app; this creates a booking record on the platform.
✓  App shows event summary stats: total registered, checked-in count, and remaining expected attendees.
Vibe Code Brief
Build with React Native. Single app with role-based navigation: on login, API returns user.role (staff | attendee | admin) and routes to the appropriate stack. Staff stack: EventList screen → AttendeeList screen → optionally QRScanner (FEAT-16). API endpoints: GET /app/events (staff upcoming events), GET /app/events/:id/attendees (list with checkin_status), POST /app/checkins (create or toggle check-in), POST /app/walkins (create booking + immediate check-in). Offline: use @react-native-async-storage/async-storage to queue POST /app/checkins payloads when network is unavailable; use NetInfo from @react-native-community/netinfo to detect connectivity changes and flush the queue on reconnect.
